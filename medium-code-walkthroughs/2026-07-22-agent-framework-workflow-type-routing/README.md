# Microsoft Agent Framework: Type-Based Routing and Pregel Supersteps

Runnable code for the Medium article *"Microsoft's Agent Framework Routes Workflow
Messages by Python Type — and Runs Them in Pregel Supersteps"* (2026-07-22).

Both examples run locally, CPU-only, **with no LLM and no API key** — they exercise
the workflow engine itself, which is model-agnostic.

- `agent-framework-core` **1.12.0** (released 2026-07-21)

## What each file shows

### `01_type_routing.py` — an edge delivers by message type, not by wiring

A `Classifier` emits either a `Refund` or a `Question`. Two downstream nodes are wired
with identical edges: `RefundDesk` handles both types, `FAQ` handles only `Question`.
The trace proves a `Refund` never reaches `FAQ`.

Real captured output:

```text
refund_desk handles: ['Question', 'Refund']
faq         handles: ['Question']

[Refund ticket] -> Refund is a type only RefundDesk handles
  RefundDesk.on_refund invoked
outputs: ['RefundDesk paid $42']

[Plain question] -> Question is a type BOTH downstream nodes handle
  RefundDesk.on_question invoked
  FAQ.on_question invoked
outputs: ['RefundDesk relayed: What are your opening hours?', 'FAQ answered: What are your opening hours?']
```

### `02_supersteps.py` — the Pregel/BSP superstep model

A `Dispatch` node fans out to `fraud` and `stock` checks, which fan in to `approve`.
Grouping the streamed `executor_invoked` events by superstep shows fan-out nodes share
one superstep (concurrent) and the fan-in node runs in the next, after the barrier.

Real captured output:

```text
OUTPUT: APPROVED with fraud=ok(A1001) & stock=ok(A1001)
superstep 1: ran ['fraud', 'stock']
superstep 2: ran ['approve']
```

> Note: the order of `'fraud'` and `'stock'` *within* superstep 1 is not guaranteed
> (they run concurrently); the superstep *grouping* is what matters and is stable.

## How the claims were verified

Beyond running the code, the article's two structural claims were confirmed by reading
the installed 1.12.0 source directly (not the docs):

- `agent_framework/_workflows/_runner.py` — the runner class docstring reads
  *"A class to run a workflow in Pregel supersteps."*
- `agent_framework/_workflows/_edge_runner.py` — each delivery is gated on
  `self._can_handle(edge.target_id, message)`.
- `agent_framework/_workflows/_executor.py` — `Executor.can_handle` returns
  `any(is_instance_of(message.data, message_type) for message_type in self._handlers)`.
